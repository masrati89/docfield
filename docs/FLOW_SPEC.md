# FLOW_SPEC.md — inField

## מסמך Flow מלא לביצוע על ידי Claude Code

> **מסמך זה הוא מקור האמת לכל לוגיקת ה-Flow באפליקציה.**

## §1. היררכיית ניווט

```
App
├── Auth (Login / Register)
└── Main (Tab Bar — 4 tabs)
    ├── בית (Home Dashboard)
    ├── דוחות (Reports List)
    ├── פרויקטים (Projects List)
    │   └── Project → Buildings → Apartments → Report
    └── הגדרות (Settings)
```

**כלל ניווט:** כל כניסה לדוח יכולה לבוא משני נתיבים:

1. **נתיב פרויקט:** בית → פרויקטים → בניין → דירה → דוח (קונטקסט מלא)
2. **נתיב ישיר:** בית / דוחות → לחיצה על דוח קיים → דוח (קונטקסט מדוח)

## §2. יצירת פרויקט חדש

### כללי יסוד

- סוג הפרויקט (`delivery` / `bedek_bait`) נקבע **בהקמה** ומוגדר כ-default לכל הדוחות בפרויקט.
- ניתן לשנות את ה-default **רק בהגדרות הפרויקט** ורק לדוחות **עתידיים** — דוחות קיימים לא משתנים לעולם.
- **תבנית הצ'קליסט** לפרויקט `delivery` נבחרת גם בהקמה. ניתן לשנות בהגדרות, גם כן רק לדוחות עתידיים.

### Add Project Sheet — שדות

```
שם פרויקט (required)
כתובת (required)
──── סוג פרויקט (required) ────
  ○ פרוטוקול מסירה  ● בדק בית   ← pill buttons
──── אם נבחר "פרוטוקול מסירה" ────
  תבנית דירה ברירת מחדל:
  [ דירת 3 חדרים ▼ ]   ← dropdown
  אפשרויות: 3 חדרים / 4 חדרים / 5 חדרים / דירת גן / קוטג' / מותאם אישית
────────────────────────────────
[ צור פרויקט ]
```

### UX Impact לפי סוג פרויקט

|                        | `delivery`                                    | `bedek_bait`                      |
| ---------------------- | --------------------------------------------- | --------------------------------- |
| FAB "בדיקה חדשה" בדירה | פותח ישר פרוטוקול (ללא בחירת סוג)             | פותח ישר בדק בית                  |
| שדה "סבב מסירה"        | ✅ מופיע                                      | ❌ לא מופיע                       |
| Checklist חדרים        | ✅ פעיל                                       | ❌ לא פעיל (free-form)            |
| סטטוסי דירה            | pending → in_progress → completed → delivered | pending → in_progress → completed |
| Progress tracking      | לפי דירות שנמסרו                              | לפי דוחות שהושלמו                 |

## §3. יצירת דוח חדש — Wizard Flow

### עיקרון

"בדיקה חדשה" פותחת **Bottom Sheet אחד** עם wizard של שלבים.
כל שלב מחליף את תוכן ה-Sheet בanimation slide (ימין←שמאל).
חץ חזרה ← בHeader מחזיר לשלב הקודם.
ה-Sheet לא נסגר בין שלבים.

### מבנה Wizard — 4 שלבים אפשריים

```
שלב 1 (תמיד)           → בחר סוג דוח
שלב 2 (תמיד)           → שיוך לפרויקט (אופציונלי)
שלב 3 (מותנה)          → שיוך לדירה — רק אם שויך פרויקט בשלב 2
שלב 4 (מותנה)          → מצב פרוטוקול — רק אם סוג = פרוטוקול מסירה
────────────────────────
→ התחל בדיקה
```

---

### שלב 1 — בחר סוג דוח

```
כותרת: "בדיקה חדשה"           [X סגור]
progress: ●○○○

בחר סוג דוח:

┌─────────────────────┐  ┌─────────────────────┐
│   📋 בדק בית        │  │  ✓ פרוטוקול מסירה   │
│   ליקויים חופשיים   │  │  צ׳קליסט מובנה      │
└─────────────────────┘  └─────────────────────┘
  (בהמשך: סוגים נוספים כגון דוח מפקח, איתור נזילות)

[ המשך → ]   ← disabled עד שנבחר סוג
```

**כלל:** סוג הדוח הוא הבחירה היחידה בשלב זה. לחיצה על כרטיס = בחירה ישירה (לא checkbox).

### שלב 2 — שיוך לפרויקט

```

progress: ●●○○

שייך לפרויקט? (אופציונלי)

🔍 חפש פרויקט...
────────────────
פרויקטים אחרונים:
  📁 פארק תל-אביב         →
  📁 גבעת שמואל מגדלים    →
  📁 נווה צדק              →
```

## §4. מסירה שנייה (Round 2)

### איפה כפתור "מסירה שנייה"

כפתור "מסירה שנייה" מופיע ב**שני מקומות**:

1. **ברמת הדירה** — בכרטיס הדירה במסך Apartments, כאשר `apt.status === 'completed'` ויש דוח ראשון מושלם
2. **ברמת הדוח** — בתוך הדוח הראשון (Checklist screen), בtoolbar הדוח — כפתור "מסירה שנייה"

### Flow — יצירת מסירה שנייה

```
לחיצה "מסירה שנייה"
→ Confirmation dialog:
  "ליצור פרוטוקול מסירה שנייה לדירה זו?
   הליקויים מהמסירה הראשונה יועברו לבדיקה."
  [ביטול]  [צור מסירה שנייה]

→ המערכת:
  1. יוצרת delivery_report חדש (round_number=2, previous_round_id=round1.id)
  2. מעתיקה ממצאים שסומנו כ-"לא תקין" / "תקין חלקית" מהסבב הראשון
  3. כל ממצא מקבל review_status = 'pending_review'
  4. פותחת את הדוח החדש

→ בדוח הסבב השני:
  כל ממצא שהועבר מוצג עם badge "מסבב 1" + סטטוס review:
  pending_review / fixed / not_fixed / partially_fixed
```

### כלל: מסירה שנייה קיימת?

אם כבר קיימת מסירה שנייה לדירה — **הכפתור לא מוצג**. במקומו מוצגת קישור "צפה במסירה שנייה".

## §5. סטטוסי דוח — Flow מלא

### 4 סטטוסים

| Status DB     | תווית  | צבע                  | מתי                             |
| ------------- | ------ | -------------------- | ------------------------------- |
| `draft`       | טיוטה  | cr200 bg, n600 text  | ברגע יצירת הדוח                 |
| `in_progress` | בעבודה | go100 bg, go700 text | המפקח עובד על הדוח (שינוי ידני) |
| `completed`   | הושלם  | g50 bg, g700 text    | המפקח סימן "הושלם" + אישור כפול |
| `sent`        | נשלח   | g100 bg, g700 text   | הדוח נשלח ב-WhatsApp            |

### מעברי סטטוס מותרים

```
draft ──── (ידני: "סמן בעבודה") ────→ in_progress
draft / in_progress ── ("סמן כהושלם" + אישור כפול) ──→ completed
completed ── ("עריכה" + אישור כפול) ──→ draft
completed / sent ── (שליחה ב-WhatsApp) ──→ sent
```

**כלל עריכה:** ניתן לערוך רק `draft` ו-`in_progress`. עריכת `completed` דורשת אישור כפול וחזרה ל-`draft`.

### Confirmation Dialogs

**סימון "הושלם":**

```
"לסמן את הדוח כהושלם?
 לאחר מכן ניתן לערוך רק עם אישור נוסף."
[ביטול]  [סמן כהושלם]
```

**חזרה לעריכה מ-completed:**

```
"הדוח הושלם. לחזור למצב עריכה?
 יהיה צורך לסמן אותו כהושלם שוב לאחר העריכה."
[ביטול]  [אישור]
```

## §6. סטטוסי דירה (Apartment Status)

### פרויקט מסירה (`delivery`)

| ------------- | ------ | ----- | -------- | ---------------------------- |
| `pending`     | ממתין  | n400  | cr200 bg | טרם נפתח דוח                 |

## §7. בדק בית — Flow פנימי

### מסך MainScreen (Tab ממצאים)

**4 טאבים:**

- `ממצאים` — רשימת קטגוריות + ממצאים
- `פרטי דוח` — שם דוח, מזמין, תאריך, כתובת, הערות
- `תוכן` — תבנית הדוח (סדר קטגוריות, כותרות)
- `חוסרים` — ממצאים שחסרים להם: תקן / המלצה / עלות

**כלל badge "חוסרים":** Badge אדום תמיד, גם כשטאב לא פעיל. מציג מספר ממצאים עם שדות חסרים.

### Entry Paths להוספת ממצא

| Entry                         | מה קורה                      |
| ----------------------------- | ---------------------------- |
| Footer "הוסף ממצא"            | טופס ריק, בחר קטגוריה תחילה  |
| "+" בקטגוריה                  | טופס עם קטגוריה pre-selected |
| Camera flow → "הוסף ממצא (X)" | טופס עם תמונות pre-attached  |
| מאגר ממצאים → בחר פריט        | כל השדות ממולאים             |

### Camera Flow (מפורט)

```
לחיצה על 📷 בfooter
→ Camera overlay (מסך מלא, רקע שחור)
  - X למעלה ימין (סגור ובטל)
  - Badge ירוק עם מונה תמונות (למעלה שמאל) — מופיע אחרי צילום ראשון
  - Shutter button (68px עגול, border לבן)

→ צילום:
  - תמונה נוספת למערך זמני
  - עובר אוטומטית ל-Review mode

→ Review mode:
  - כותרת "X תמונות צולמו"
  - גריד 96×96 עם X לכל תמונה
  - [צלם עוד] → חזרה ל-viewfinder
  - [הוסף ממצא (X)] → פותח AddDefect עם תמונות pre-attached

כלל: סגירת Camera ב-X מבטלת את כל התמונות שצולמו.
כלל: לא ניתן לצאת ב-Camera ולשמור תמונות בלי לפתוח ממצא.
```

### Swipe-to-Delete (רק בבדק בית)

```
swipe שמאל על ממצא → מחשיף כפתור מחיקה אדום (80px)
לחיצה על מחיקה → Confirmation: "למחוק ממצא זה?"
                    [ביטול]  [מחק]

First-time hint: אנימציית nudge עם "החלק ←" על הממצא הראשון בקטגוריה הראשונה
```

### Search Overlay — בדק בית

```
Bottom sheet 75vh
↑ שורת chips לפילטר קטגוריה: [הכל] [אינסטלציה] [חשמל] ...
↑ Input חיפוש (fontSize 16, dir="rtl")
↑ "חיפושים אחרונים" + רשימה עם icon שעון
Exit: slideDown + fadeOut
```

## §8. פרוטוקול מסירה — Flow פנימי

### מסך Checklist

**כלל accordion:** חדר אחד פתוח בכל פעם. פתיחת חדר חדש סוגרת את הפתוח.

**כלל "דלג":** פריט שסומן "דלג" חוזר לסקירה בסוף הצ'קליסט. רק שם ניתן לסמן "לא רלוונטי" עם אישור כפול.

**כלל defect detail row:** רק שורת פירוט ליקוי אחת פתוחה בכל פעם.

**כלל Bath Type:** selector "מקלחון / אמבטיה" מופיע בראש חדרי רחצה. הבחירה משפיעה על הפריטים הגלויים.

**כלל conditional items:** פריטים עם `parentId` מוצגים רק כאשר הסטטוס של הפריט ההורה = "ok".

### 4 סטטוסי Checklist

| Status     | Symbol | Color             | Background | Border  |
| ---------- | ------ | ----------------- | ---------- | ------- |
| תקין       | ✓      | g500 (#1B7A44)    | #ecfdf5    | g200    |
| לא תקין    | ✗      | clRed (#b91c1c)   | #fef2f2    | #f5c6c6 |
| תקין חלקית | ~      | clAmber (#92600a) | #fefaed    | #f0dca0 |
| דלג        | —      | n400              | cr100      | cr300   |

**חשוב:** `clRed` לצ'קליסט, `sR` (#EF4444) לשגיאות מערכת בלבד.

### FAB Sheet — הוספת ליקוי (Checklist)

3 שדות בלבד:

1. **קטגוריה** (required) — ComboField, רשימה: אינסטלציה / חשמל / אלומיניום / ריצוף / טיח וצבע / איטום / נגרות / כללי
2. **מיקום** (optional) — ComboField, רשימה מהחדרים בדוח הנוכחי
3. **תיאור הליקוי** (required) — textarea, fontSize 16, dir="rtl"

CTA: "הוסף ליקוי" — disabled עד שקטגוריה + תיאור מולאו.

## §14. Iron Rules — אסור לשנות לעולם

1. **דוח הוא מסמך משפטי.** שום שינוי במערכת לא משנה דוח קיים.
2. **Defects כ-Snapshot.** ממצאים נשמרים כטקסט מלא — לא כ-FK לмאגר. מחיקת פריט ממאגר לא שוברת דוח.
3. **PDF אחד.** לא גרסאות. דורס.
4. **Activity Log** מתחיל מ-PDF ראשון בלבד.
5. **חתימות** — immutable. אין UPDATE policy ב-DB.
6. **RLS על כל טבלה.** ללא יוצא מהכלל.
7. **Tenant isolation** — `organization_id` על כל query.
8. **סוג פרויקט** — שינוי בהגדרות משפיע רק על דוחות עתידיים.

## §15. UX Rules — מתוך המוקאפים

### כלל iOS Zoom Prevention

כל `input` ו-`textarea` חייב: `fontSize: 16`, `autoComplete="off"`, `spellCheck={false}`

### כלל Bottom Sheet

- Backdrop: `rgba(0,0,0,.4)`
- Animation IN: `slideUp 0.35s cubic-bezier(.22,1,.36,1)`
- Animation OUT: `setClosing(true) → 250ms → unmount`
- תמיד: animate-out לפני unmount

### כלל Dropdown

רק dropdown אחד פתוח בכל פעם. לחיצה מחוץ — סגור הכל.

### כלל אנימציית כניסה לרשימות

Staggered entrance: `60ms delay` לכל פריט.
`riseIn` animation: `opacity 0→1 + translateY(12px→0)`.

### כלל Loading

Skeleton screens בכל מסך שמביא נתונים. אין spinners.

### כלל Empty State

כל רשימה ריקה מציגה: emoji + כותרת + תיאור + CTA.

### כלל Dirty Check

כל טופס עם נתונים לא שמורים שמשתמש מנסה לצאת ממנו:

```
"יש שינויים שלא נשמרו. לצאת בלי לשמור?"
[ביטול]  [צא בלי לשמור]
```

### כלל Safe Area

כל footer קבוע: `padding-bottom: max(24px, env(safe-area-inset-bottom))`

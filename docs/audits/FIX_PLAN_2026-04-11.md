## 2. קיבוץ תיקונים לפי root cause

### 🅐 קבוצה A — נתיב ה-PDF והתצוגה המקדימה

**באגים:** 01, 02, 05, 06, 04-חלקי

**הבעיה האמיתית:**

- `expo-print` ב-web **לא תומך** ב-`Print.printToFileAsync`. זו הסיבה ש"הפק PDF" נכשל בשקט אחרי הרענון — הקריאה זורקת שגיאה, ו-`catch {}` ב-`usePdfGeneration.ts:285` בולע אותה לגמרי. צריך ענף `Platform.OS === 'web'` + החלפת `catch {}` ב-log אמיתי.
- `handleGeneratePdf` ב-`app/(app)/reports/[id]/index.tsx:207` נקרא גם מ-preview וגם ישירות. כשנקרא מ-preview הוא לא סוגר אותו לפני פתיחת הסאמרי → שני modals חופפים. הפתרון — לא zIndex, אלא `setShowPreview(false)` לפני `setShowSummary(true)`.
- `previewHtml.ts:341,441` משווה `data.property.floor !== null` — אבל `floor` מגיע `undefined` כי (א) `useReport` לא מושך אותו מה-DB, (ב) `buildPreviewData` ב-`ReportPreviewModal.tsx:44-49` לא מעביר אותו בכלל. גם הטיפוס ב-`lib/pdf/types.ts` צריך לאפשר `undefined`.
- `previewHtml.ts:344,366,436` מדפיס `${escapeHtml(data.reportDate)}` ללא פורמט.
- `sharePdf` ב-`usePdfGeneration.ts:311-314` מחזיר error על web. צריך להוסיף ענף web שמשתמש ב-`navigator.share` (אם זמין) או fallback ל-print dialog.

**קבצים שישתנו:**

- `apps/mobile/hooks/usePdfGeneration.ts` — ענף web ב-`generatePdf` + ב-`sharePdf`, הסרת `catch {}` אילם, הוספת console.error
- `apps/mobile/app/(app)/reports/[id]/index.tsx` — בין preview ל-summary: closing logic (סגירת preview לפני פתיחת summary)
- `apps/mobile/lib/pdf/previewHtml.ts` — שורות 341, 441: `!== null` → `!= null`; שורות 344, 366, 436: עטיפת `reportDate` ב-formatter
- `apps/mobile/components/reports/ReportPreviewModal.tsx` — `buildPreviewData` מעביר `floor`
- `apps/mobile/hooks/useReport.ts` — הוספת `floor` ל-`ReportInfo` + ל-SELECT query
- `apps/mobile/lib/pdf/types.ts` — אם `PdfReportData.property.floor` מוגדר `number | null`, להרחיב ל-`number | null | undefined`

**גישה נבחרה ולמה:**

- **PDF ב-web:** שימוש ב-`Print.printAsync({ html })` במקום `printToFileAsync`. `printAsync` על web פותח חלון הדפסה של הדפדפן → משתמש יכול לשמור כ-PDF דרך הדיאלוג. זה API יציב, מוכר למשתמש Chrome/Safari, וללא תלות חדשה. חלופה נשקלה (יצירת Blob ו-`URL.createObjectURL`) אבל דורשת lib נוספת — YAGNI.
- **z-index/preview:** העדיפות — לסגור preview לפני summary ולא להסתבך עם stacking contexts. אפילו יותר פשוט מ-zIndex: 1000.
- **floor:** שלוש הנקודות (DB fetch → preview data builder → HTML template) חייבות להיות בסינק. `!= null` (loose) הוא תיקון זול שמכסה גם את המקרה העתידי בו floor באמת null.
- **share-on-web:** מימוש `navigator.share` על הדפדפנים המודרניים. נופל חזרה ל-`printAsync` עם toast "שמור דרך דיאלוג ההדפסה".

**סיכוני רגרסיה:**

- `Platform.OS === 'web'` branching בתוך hook שמשותף ל-native ול-web — צריך לוודא שה-native path נשאר 100% לא נוגע.
- הסרת `catch {}` תגרום ל-errors לצוף ב-console בצורות שהמשתמש לא היה רגיל אליהן. זה רצוי, אבל יכול להיראות כ"רגרסיה" לעין לא מיומנת.
- `useReport` משותף ל-~10+ קומפוננטות. הוספת `floor` לתוצאה היא תוספת אחורה-תואמת, לא תשבור כלום, אבל להיות ערניים.
- ה-Iron Rule לגבי snapshot — שימו לב: אנחנו לא מכניסים live join חדש ל-PDF, אנחנו רק מתקנים את ה-**preview** (לא PDF). `usePdfGeneration.ts` כבר מכבד את ה-Iron Rule ולא נוגעים בזה. אל תיתן ל-`floor` להגיע ל-`lib/pdf/bedekBayit.ts` או `protocol.ts` דרך live fetch — הוא כבר מגיע משם דרך `property_floor` snapshot column, בסדר.

**סדר ביצוע בתוך הקבוצה:**

1. 🔹 A.1 — `useReport.ts`: הוסף `floor: number | null` ל-`ReportInfo` + SELECT query
2. 🔹 A.2 — `ReportPreviewModal.tsx`: `buildPreviewData` מעביר `floor: report.floor ?? undefined`
3. 🔹 A.3 — `previewHtml.ts`: `!= null` + עטיפת `reportDate` ב-formatter פנימי
4. 🔹 A.4 — `usePdfGeneration.ts`: web branch ב-`generatePdf` (`Print.printAsync` במקום `printToFileAsync` + log שגיאות)
5. 🔹 A.5 — `usePdfGeneration.ts`: web branch ב-`sharePdf` (`navigator.share` + fallback)
6. 🔹 A.6 — `app/(app)/reports/[id]/index.tsx`: `handleGeneratePdf` נקרא מ-preview → סגור preview קודם

**אימות אחרי הקבוצה:**

- `npm run typecheck` — אפס שגיאות
- Playwright תרחיש: דוח → "הורדת דוח" → "הפק PDF" → לוודא שדיאלוג הדפסה נפתח
- Playwright תרחיש: דוח → "תצוגה מקדימה" → בתוך ה-preview "הפק PDF" → לוודא שה-preview נסגר, summary נפתח למעלה, ולאחר "הפק PDF" שוב נפתח דיאלוג הדפסה
- בדיקה ויזואלית של preview: אין "קומה undefined", אין ISO גולמי

## 3. סיכוני רגרסיה כלליים (חוצי קבוצות)

1. **Multi-tenant/RLS:** לא נוגעים ב-select queries מחוץ לתוספת `floor` ב-`useReport`. ה-`floor` הוא שדה של `apartments` שכבר נטען — הוא רק לא נחשף ב-TS. אפס impact ב-RLS.
2. **Auth flow:** לא נוגעים ב-AuthContext או ב-gotrue. השגיאה המקורית של "Lock broken" שראיתי בריצה הראשונה עשויה להיעלם אוטומטית ברגע שנחליף `printToFileAsync` ל-`printAsync` (כי ה-native flow ארוך וסוגר את ה-async window שבו gotrue מאבד lock). אם לא — זה תיקון עתידי נפרד.
3. **Iron Rule (Snapshot-only PDFs):** הוספת `floor` ל-`useReport` היא **לא** הפרה — היא נוגעת ל-**preview** בלבד. `PdfReportData.property.floor` ב-PDF עצמו כבר מגיע מ-`property_floor` snapshot column (`usePdfGeneration.ts:211`). לא להוסיף live join לשום PDF generator (bedekBayit.ts / protocol.ts) גם אם מתפתים.
4. **Platform.OS branches:** שינויים ב-`usePdfGeneration` חייבים לא לגעת ב-native path — לעטוף את החדש ב-`if (Platform.OS === 'web')` ולא להחליף את ה-default.
5. **React 19 vs 18:** התיקון ל-BUG-08 מניח React 18. אם הפרויקט משדרג ל-19 בעתיד, ה-`forwardRef` ימשיך לעבוד (תאימות אחורה מובטחת).

## 6. ממצאים נוספים (לא בתוך scope — לא לתקן עכשיו, רק לרשום)

במהלך החקירה גיליתי את הדברים האלה; **לא אתקן אותם** בסבב הזה לפי הוראה מפורשת:

1. **`usePdfGeneration.ts:285` — `catch {}` אילם:** בולע גם את השגיאה מ-`printToFileAsync` וגם כל שגיאה אחרת בצינור הפקת PDF. גם אם נתקן את ה-web path, ה-native path עדיין בולע שגיאות. רציתי להוסיף `console.error(err)` מינימלי בתוך התיקון של A.4 — זה לא הרחבת scope, זה הגיינה שמלווה את התיקון. **מבקש אישור להוסיף את ה-log גם ל-generic catch, לא רק ל-web branch.**

2. **Double-fetch ב-`ReportPreviewModal`:** הקומפוננטה קוראת ל-`useReport(reportId)` באופן עצמאי למרות שה-parent כבר קרא לו. React Query cache מגן מפני רשת כפולה, אבל זה יוצר re-render מיותר. אופטימיזציה עתידית, לא באג.

3. **`reports/[id]/index.tsx:204`** — `subtitle = report ? `${report.address ?? report.projectName}, דירה ${report.apartmentNumber}` : ''`. אם `apartmentNumber` ריק → נוצר `", דירה "` עם פסיק יתום. זה **BUG-18** ב-QA report (רמה 🔵 "טקסט/דקדוק"). מחוץ ל-P0/P1 scope.

4. **`add-defect.tsx:666`** — `onBlur` של description מסתיר suggestions אחרי 200ms. על web ה-200ms לא תמיד מספיק לקליק → לפעמים ה-suggestion לא נלחצת. זה באג אמיתי שלא היה בדוח — **אני רושם אותו כ-BUG-24 חדש** למעקב עתידי, לא לתקן עכשיו.

5. **`checklist.tsx` לא נבדק**: אימתתי רק שהוא לא משתמש ב-`ReportTabBar`. לא בדקתי את שאר ההתנהגות.

## 7. נספח — הצדקות לדחיות false positives

### BUG-07 (CategoryPicker state) — דחוי

**טענה מקורית:** "בחירת קטגוריה לא מתמשכת אחרי שמירה + חזרה — `resetForm()` שנקרא אחרי `onBlur`"
**ממצא:** `add-defect.tsx:71` משתמש ב-`ComboField` לא ב-`CategoryPicker`. הקובץ כולו לא מכיל `resetForm` — הוא רק קורא ל-`router.back()` אחרי שמירה (שורה 428). המצב נמחק בניווט החוצה, לא ב-blur. אין באג ב-state management.

### BUG-09 (Defect library selection) — דחוי

**טענה מקורית:** "בחירת פריט מהמאגר לא מעדכנת שדות"
**ממצא:** `handleSelectSuggestion` ב-`add-defect.tsx:128-143` מזין את כל 5 השדות: description, category, location, standardRef, recommendation. הוא משתמש בהגנה `if (item.X && !currentX)` כדי לא לדרוס ערכים קיימים — זה פיצ׳ר לא באג. אם ה-QA ראה שהשדה לא התעדכן, סביר שהוא היה ערך קיים קודם (או שה-suggestion panel נסגר לפני הקליק — ראה ממצא #4 בסעיף 6).

### BUG-10 (Shortages count = 8) — דחוי

**טענה מקורית:** "דירה חדשה מראה 8 חוסרים לפני שבוצע checklist"
**ממצא:** `useReportShortages.ts` בונה 5 קטגוריות חוסר־חובה:

- פרופיל בודק (3 required: full name, license, signature)
- פרטי לקוח (2 required: name, phone)
- פרטי נכס (3 required: project, address, apartment)
- תוכן הדוח (1 required: declaration)
- ממצאים (1 required: at least one)

סה״כ 10 אפשריים. דוח חדש עם רק `full_name` ו-`apartment_number` → 8 חסרים. **זה נכון**. הטאב "חוסרים" לא קשור ל-checklist template — הוא מנוע שלמות דוח, וסימון "8" הוא התנהגות מלאה ונכונה.

### BUG-13 (4-tab bar on delivery) — דחוי

**טענה מקורית:** "Tab bar 4-tabs נראה גם ב-delivery reports"

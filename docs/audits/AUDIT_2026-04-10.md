# inField — Audit Report

**Date:** 2026-04-10
**Scope:** Read-only audit (no modifications). Focus on red flags, architectural consistency, and unfinished features.
**Baseline:** Builds on prior `AUDIT_REPORT.md` (2026-04-09, grade B+). This report targets what that one missed, especially Iron Rule enforcement and RLS gaps.

---

## Part 1 — Overview

**Product.** Hebrew/RTL construction inspection SaaS for Israel. Two report modes: `bedek_bait` (defect inspection) and `delivery` (protocol mesira). Competes with Reporto on native feel + offline capability.

**Stack.** Turborepo monorepo. Mobile: Expo SDK 55 + React Native 0.76.9 + Expo Router v4 + NativeWind + React Query 5. Backend: Supabase (Postgres + Auth + Storage + Edge Functions). Shared packages (`@infield/shared`, `@infield/ui`). Web app exists but is secondary.

**State of the codebase.** Phase 6. Most screens built and refactored after a prior cleanup sprint (three giant files reduced: 1493→499, 1320→297, 1105→247). 31 SQL migrations (CLAUDE.md still says 15). 20 hooks (CLAUDE.md says 13). Library screen (450 lines) is fully built though CLAUDE.md lists it "not built yet". A Supabase Edge Function `sync` exists but is a TODO stub.

**Iron Rule status (the headline finding).** Partially broken. Migration 029 added 16 snapshot columns (inspector + org) and `createReportWithSnapshot.ts` writes them correctly. BUT:

1. **Project / building / apartment data is not snapshotted** — PDFs still read live joined data.
2. **Inspector signatures are stored at a fixed path with `upsert: true`** — updating the signature silently rewrites every old PDF that links to it.
3. **RLS has zero status-based enforcement** — nothing at the DB level prevents defects/checklist/report edits on `completed` reports; protection is client-side only.

**Flows.** Auth (email + Google + Apple OAuth) → register creates org + profile (3 hard-coded steps, all users become `admin`) → Home dashboard → New Inspection Wizard (6 steps) → Report detail → Checklist / Add Defect → PDF (expo-print) → share. Offline sync is not implemented.

## Part 2 — Findings (classified by severity)

### 🔴 Critical

#### C1. Signature/stamp files use `upsert: true` on fixed paths — destroys old PDFs

- **Location:** `apps/mobile/hooks/useSignature.ts:40-78` (inspector), `:96-130` (stamp).
- **Description:** Path is `${organizationId}/inspector_${profile.id}.png`; upload uses `upsert: true`. When the inspector updates their signature or stamp, the storage object is overwritten in place. Every previously issued PDF stores `inspector_signature_url_snapshot` / `inspector_stamp_url_snapshot` pointing to that same URL → the new signature silently replaces the signature on every historical report.
- **Impact:** Iron Rule violation on a legal document. Worst-case: a dispute shows a defendant's "signature" that never actually existed at report-signing time. `deleteInspectorSignature` (lines 160+) makes it worse — old reports then 404.
- **Effort:** S — write to a new content-hashed or timestamped path each upload; signatures are immutable per migration 009 intent.

#### C2. PDF generation reads live project/apartment data instead of snapshot

- **Location:** `apps/mobile/hooks/usePdfGeneration.ts:244-245` (`projectName: apt?.buildings?.projects?.name`, `address`, `apartmentNumber`).
- **Description:** Inspector + org fields correctly use snapshot columns, but project name, project address, building name, and apartment number are pulled via live joins on `apartments → buildings → projects` at PDF-generation time. If a project is renamed or an apartment is deleted between report close and PDF re-generation, the PDF changes.
- **Impact:** Iron Rule violation. Affects the most visible header section of every report.
- **Effort:** M — add snapshot columns in a new migration (`project_name_snapshot`, `project_address_snapshot`, `building_name_snapshot`, `apartment_number_snapshot`), write them in `createReportWithSnapshot.ts`, read them in `usePdfGeneration.ts`.

#### C3. RLS has no status-based protection on completed reports

- **Locations:**
  - `supabase/migrations/007_create_defects.sql:54-65` — `defects_update` has zero status check.
  - `supabase/migrations/005_create_delivery_reports.sql` + later — `delivery_reports_update` never checks the transition from `completed` → anything.
  - `supabase/migrations/006_create_checklist_results.sql` — same, no status guard.
- **Description:** The Iron Rule ("Completed reports require double confirmation to re-edit") is enforced **only** in the client (`useReportStatus.ts:92-126`). Any authenticated same-org user with a valid session token can `UPDATE` defects or reports on a completed report via the Supabase client directly. Even the app bypasses its own guard in one path: `apps/mobile/app/(app)/reports/[id]/index.tsx:313-318` calls `.update({ status: newStatus })` with no confirmation.
- **Impact:** Legal-document integrity is not defensible. A malicious or buggy client can silently mutate closed reports.
- **Effort:** M — add `USING (... AND status != 'completed')` to update/delete policies on `delivery_reports`, `defects`, `checklist_results`. Status transitions must go through a `SECURITY DEFINER` function that enforces the double-confirmation server-side.

#### C4. `defects_insert` RLS does not verify the parent report belongs to the caller

- **Location:** `supabase/migrations/007_create_defects.sql` (defects_insert policy).
- **Description:** Policy checks only `organization_id = get_user_org_id()`. An authenticated user can insert a defect into _any_ report in their org regardless of who owns it, including a completed one. Combined with C3, a bad actor can backfill defects into closed legal reports.
- **Impact:** IDOR within the organization; Iron Rule bypass.
- **Effort:** S — add `EXISTS (SELECT 1 FROM delivery_reports r WHERE r.id = delivery_report_id AND r.inspector_id = auth.uid() AND r.status != 'completed')` to the `WITH CHECK`.

#### C5. `sync` Edge Function is a TODO placeholder but is wired into the product narrative

- **Location:** `supabase/functions/sync/index.ts:117-129`.
- **Description:** The file returns an empty changeset and is marked TODO. It also uses the anon key instead of the service role. CLAUDE.md lists offline sync as "deferred post-MVP" but the endpoint is deployed and callable.
- **Impact:** Any client assuming "sync succeeded" will silently lose data. Auth is also wrong for a server-side sync endpoint.
- **Effort:** Either delete the function entirely or implement it. Recommended: delete until needed.

### 🟠 High

#### H1. `028_rls_recursion_fix` only fixed the `users` table — other policies still use the inline subquery

- **Location:** `supabase/migrations/028_rls_recursion_fix.sql` vs most other migrations (003–014).
- **Description:** Migration 028 introduced `get_user_org_id()` (SECURITY DEFINER) but rewrote only the `users` policies. Every other table (`projects`, `delivery_reports`, `defects`, `signatures`, `checklist_results`, `notifications`, etc.) still embeds `organization_id = (SELECT organization_id FROM users WHERE id = auth.uid())`. This works today only because 028 broke the recursion on `users`, but the pattern is inconsistent, harder to audit, and slower (subquery per row on every hot-path query).
- **Impact:** Architectural debt; any future change to `users` RLS risks reintroducing recursion across the whole DB.
- **Effort:** M — one sweep migration replacing the subquery with `get_user_org_id()` on every policy.

#### H2. Every registered user becomes `admin`

- **Location:** `apps/mobile/app/(auth)/register.tsx:231` (`role: 'admin'` hard-coded).
- **Description:** All users insert their profile with `role='admin'`. All role-based RLS branches (`WHERE u.role = 'admin'`) are therefore always true. "Inspector" and "project_manager" roles are dead code until team/invitation flow is wired up.
- **Impact:** Role-based authorization is a no-op. Migration 029's `organization_members` table with `owner/admin/project_manager/inspector` is never exercised.
- **Effort:** S — first user in an org = admin; invited users default to `inspector`. Requires the invitation flow to actually ship.

#### H3. `delivery_reports` cascade-deletes to `signatures` despite signatures being "immutable"

- **Location:** `supabase/migrations/009_create_signatures.sql` (FK `ON DELETE CASCADE`).
- **Description:** The migration comments "NO UPDATE POLICY" / "NO DELETE POLICY" to enforce immutability, but deleting the parent report bypasses those policies via cascade. Admins can hard-delete reports, which also deletes the signatures — leaving no forensic trail.
- **Impact:** Iron Rule immutability is only surface-deep.
- **Effort:** S — change FK to `ON DELETE RESTRICT` and soft-delete reports instead.

#### H4. `add-defect.tsx` is 994 lines, swallows errors, has no transaction across defect + photo uploads

- **Location:** `apps/mobile/app/(app)/reports/[id]/add-defect.tsx`.
- **Description:**
  - Exceeds the 200-line component rule by 5×.
  - Lines 319–333 contain empty `catch {}` blocks that silently ignore Supabase errors.
  - Lines 376–424: defect row is inserted first, then photos upload one by one. A mid-upload failure leaves an orphan defect with partial photos and no rollback.
  - Line 384 generates photo filenames from `Date.now() + '_' + i` — collision risk on rapid batches.
  - No numeric validation on `costAmount` / `costQty` before `parseFloat`.
- **Impact:** Silent data loss; inconsistent state; hardest screen in the codebase to modify safely.
- **Effort:** M — split into sub-hooks + extract `DefectForm` / `PhotoUploader`; wrap insert+upload in a single "promise all with rollback on failure" helper.

#### H5. `useChecklist` persists on every keystroke with no debounce, no status guard

- **Location:** `apps/mobile/hooks/useChecklist.ts:158-168`.
- **Description:** Fire-and-forget `.upsert` on every note-field keystroke. Hammers the DB, burns rate limits, and can persist edits to a completed report because it never checks `report.status`.
- **Impact:** Performance + Iron Rule (C3 expressed again in the write path).
- **Effort:** S — debounce 400ms + early-return if `report.status === 'completed'`.

#### H6. Registration flow commits inconsistent state on partial failure

- **Location:** `apps/mobile/app/(auth)/register.tsx` (steps 1→3).
- **Description:** Three independent writes: auth.signUp → insert org → insert profile. Client-generated `crypto.randomUUID()` for the org. The file has a rollback on step 3 that deletes the org and signs out, but does NOT undo the auth.signUp — the Supabase Auth user remains orphaned and cannot retry with the same email.
- **Impact:** Support burden; locked-out emails after any transient failure.
- **Effort:** M — move the whole registration to a Postgres function or Edge Function that runs as a transaction.

## Part 4 — Architectural inconsistencies

1. **Two Iron Rule implementations.** Snapshot-based for inspector/org data (correct pattern), live-join-based for project/apartment data (C2). The codebase does not have a single source of truth for "what was frozen at close time."

2. **RLS pattern drift.** Migration 028 rewrote only `users`. All other tables still use the old inline subquery. The helper exists but isn't used consistently (H1).

3. **Iron Rule enforced in three different layers — none of them at the DB.** Client hook (`useReportStatus`), screen code (double-confirmation modal), and a bypass path in `reports/[id]/index.tsx:313-318`. The DB itself is unaware of the concept of a "completed" report.

4. **Signature immutability is claimed but not enforced.** Migration 009 comments "NO UPDATE / NO DELETE" policies — correct for direct table access — but (a) `upsert: true` storage uploads sidestep this, and (b) `ON DELETE CASCADE` from reports deletes signatures via the parent.

5. **`role`-based RLS is dead code.** Policies check `u.role = 'admin'`, `register.tsx` hard-codes `'admin'`. Either rip out the checks or ship the invitation flow.

6. **CLAUDE.md drift.** The file that every session loads is significantly out of step with reality. Two identical copies (`/CLAUDE.md` and `/.claude/CLAUDE.md`) compound the problem.

7. **Report-data hook fragmentation.** `useReport` / `useReportContent` / `useReportShortages` — unclear which owns what. Low severity but likely to cause a future fix to land in only one of them.

## Part 5 — Top 3 recommendations (ranked by ROI)

### 1. Close the Iron Rule at the database layer — not the client

Covers **C1, C2, C3, C4, H3, H5, M1**. The product's entire legal positioning depends on this.

- Immutable signature storage (content-hashed paths).
- Snapshot columns for project/building/apartment; read them in PDF and in completed-report detail views.
- `USING (status != 'completed')` on update/delete policies for `delivery_reports`, `defects`, `checklist_results`.
- Parent→child deletes changed from `CASCADE` to `RESTRICT` on `signatures`.
- Status transitions through a `SECURITY DEFINER` function.

**ROI:** Fixes the single largest category of findings; turns "we claim reports are legal documents" from aspirational into defensible. One migration + one hook rewrite + storage-path change.

### 2. Sweep RLS to use `get_user_org_id()` consistently and add status guards in the same pass

Covers **H1, C3, C4**. While you're opening the policies anyway (rec #1), rewrite every `organization_id = (SELECT ...)` to `organization_id = get_user_org_id()`. One migration touches every table but mechanically — low risk, measurable perf gain, massive audit clarity.

**ROI:** Eliminates the #1 architectural drift identified. Unblocks future team/role work. Cheap if bundled with rec #1.

### 3. Update CLAUDE.md + delete the duplicate + drop the Zone.Identifier / planning-doc pollution

Covers **M2, M4**. Every new Claude session today starts with wrong file counts, wrong hook lists, wrong "not built" flags. A 20-minute fix with compounding returns on every future session.

**ROI:** Highest-leverage non-code change in the repo. Zero risk.
